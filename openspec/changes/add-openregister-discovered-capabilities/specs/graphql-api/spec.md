# Spec: graphql-api ‚Äî Unified OAS + REST‚ÜîGraph parity delta

**Status:** proposed
**Scope:** openregister
**Tier:** or-core-extensions
**Depends on:** graphql-api (existing ‚Äî extended), oas-generation (existing ‚Äî extended)

## Motivation (context for the delta)

The existing `graphql-api` spec is comprehensive: SchemaGenerator
auto-derives types from schemas, six custom scalars, query / mutation
/ subscription resolvers, RBAC integration via `PermissionHandler`
and `PropertyRbacHandler`, complexity analysis, structured error
formatting, SSE for subscriptions. Separately, `oas-generation`
covers the REST OAS document for the same schemas.

Specter's intelligence pipeline surfaced four clusters in the
"unified API surface" space (API Access, API for automation, API
Generation, Graph API). All four are individually covered by
existing infrastructure ‚Äî REST + GraphQL exist; both auto-generate
from schemas; both support all operations. **The gap is unification**:

- Today, REST consumers read one OAS document, GraphQL consumers
  read a separate introspection result. No single source describes
  both surfaces.
- There is no formal contract that REST and GraphQL serve identical
  shapes for the same schema. Divergence is possible (and observed
  in practice ‚Äî `linked-entity-types` `_extend` enrichment behaves
  differently in REST vs GraphQL because each resolver path was
  written independently).
- New consumers (n8n nodes, code generators, MCP discovery) want
  one document that lists every operation across both surfaces.

This delta adds the unified OAS 3.1 requirement and the REST‚ÜîGraph
parity guarantees. No existing GraphQL machinery is touched ‚Äî
SchemaGenerator, scalars, resolvers, RBAC integration, complexity
analysis, SSE ‚Äî all unchanged. The delta sits on top.

## MODIFIED Requirements

### Requirement: A single unified OAS 3.1 document SHALL describe both the REST and GraphQL surfaces

The system MUST publish a single OAS 3.1 document covering both the REST and GraphQL surfaces. This requirement extends the existing `graphql-api` spec
(currently silent on OAS) and the existing `oas-generation` spec
(currently REST-only). A new endpoint `GET /api/oas` MUST return
a single OAS 3.1 document covering both surfaces:

- **REST operations** under standard OAS `paths` ‚Äî generated from
  the existing `oas-generation` pipeline, unchanged.
- **GraphQL queries** under the OAS extension
  `x-graphql-queries` ‚Äî generated by walking the `SchemaGenerator`
  output and emitting one entry per root query field. Each entry
  carries `{ name, args, returnType (OAS $ref), description, security }`.
- **GraphQL mutations** under `x-graphql-mutations` ‚Äî same shape
  as queries.
- **GraphQL subscriptions** under `x-graphql-subscriptions` ‚Äî same
  shape plus a `sse: true` flag.

The shared components (`#/components/schemas/<Schema>`) MUST be the
SAME components referenced by both REST `paths` and GraphQL
operations ‚Äî one schema component, two surface references.

#### Scenario: Unified OAS document lists every schema once, referenced from both surfaces

- **GIVEN** schema `Order` registered with REST endpoints under
  `/api/objects/commerce/order` and GraphQL query `orders`
- **WHEN** `GET /api/oas` is called
- **THEN** the response MUST be a single OAS 3.1 document
- **AND** `#/components/schemas/Order` MUST be defined exactly once
- **AND** `paths./api/objects/commerce/order.get.responses.200`
  MUST `$ref` that component
- **AND** `x-graphql-queries.orders.returnType` MUST `$ref` the same
  component

#### Scenario: GraphQL queries appear under x-graphql-queries with full schema

- **GIVEN** GraphQL exposes query
  `order(id: ID!): Order` and `orders(filter: OrderFilter, first: Int): OrderConnection`
- **WHEN** the OAS document is rendered
- **THEN** `x-graphql-queries` MUST contain entries for both:
  - `order` with `{ args: [{name: "id", type: "ID", required: true}], returnType: {$ref: "#/components/schemas/Order"} }`
  - `orders` with `{ args: [{name: "filter", type: "OrderFilter", required: false}, {name: "first", type: "Int", required: false}], returnType: {$ref: "#/components/schemas/OrderConnection"} }`

#### Scenario: Subscription operations are flagged with sse: true

- **GIVEN** GraphQL exposes subscription `orderUpdated(id: ID): Order`
- **WHEN** the OAS document is rendered
- **THEN** `x-graphql-subscriptions.orderUpdated` MUST include
  `sse: true`
- **AND** MUST include a `transport: "GET /api/graphql/subscribe"` field
  pointing at the existing SSE endpoint per the existing
  `graphql-api` spec

#### Scenario: Document is valid OAS 3.1

- **GIVEN** the unified document is rendered
- **WHEN** validated against the OAS 3.1 JSON Schema
- **THEN** validation MUST pass with zero errors
- **AND** the `x-graphql-*` extensions MUST be at the document root
  (extensions allowed by OAS at any level)

### Requirement: REST and GraphQL SHALL produce identical response shapes for the same schema

REST and GraphQL MUST produce byte-comparable response shapes for the same schema. This requirement extends the existing `graphql-api` Requirement
"The GraphQL schema MUST be auto-generated from register schemas".
For every schema, the GraphQL `{schema}List` resolver and the REST
`GET /api/objects/{register}/{schema}` MUST produce responses that
agree on:

1. **Field names** ‚Äî same names, same casing convention
   (camelCase on both surfaces).
2. **Field types** ‚Äî string ‚Üî String, integer ‚Üî Int, boolean ‚Üî
   Boolean, datetime ‚Üî DateTime scalar, uuid ‚Üî UUID scalar, etc.
3. **Nullability** ‚Äî required schema fields MUST be non-nullable in
   GraphQL AND MUST always be present in REST responses (no
   silent omission).
4. **Cardinality** ‚Äî array properties MUST be lists on both
   surfaces.
5. **RBAC behaviour** ‚Äî fields hidden by `PropertyRbacHandler` in
   REST MUST also be hidden in GraphQL responses for the same
   user.
6. **Extended-type rendering** ‚Äî `calendar-range`, `recurrence`,
   `uuid`, `color`, `image-url`, etc. (per `extended-field-types`)
   MUST round-trip identically.

Parity MUST be enforced by a CI test (per Task 4.1 in this change's
tasks.md) that picks every schema, fetches the same object via
both surfaces, and asserts shape equality.

#### Scenario: REST and GraphQL agree on field names and types for a base schema

- **GIVEN** schema `Order` with properties
  `orderNumber (string), total (number), createdAt (string,
  format: date-time)`
- **WHEN** the same `Order` is fetched via
  `GET /api/objects/commerce/order/<uuid>` and via GraphQL
  `query { order(id: "<uuid>") { orderNumber total createdAt } }`
- **THEN** both responses MUST carry identical field names
  (`orderNumber`, `total`, `createdAt`)
- **AND** both MUST carry identical values
- **AND** the REST response's `createdAt` ISO 8601 string MUST be
  byte-identical to the GraphQL response's `createdAt` (`DateTimeType`
  serialises via `DateTimeInterface::ATOM`)

#### Scenario: RBAC parity hides a field consistently on both surfaces

- **GIVEN** schema `Person` has property `bsn` with
  `authorization: { "read": [{ "group": "bsn-geautoriseerd" }] }`
- **AND** user `alice` is NOT in group `bsn-geautoriseerd`
- **WHEN** `alice` fetches a `Person` via REST
- **AND** the same `alice` fetches the same `Person` via GraphQL
- **THEN** BOTH responses MUST omit the `bsn` field
- **AND** the omission MUST happen via `PropertyRbacHandler::filterReadableProperties()`
  (the single source-of-truth per `row-field-level-security`),
  invoked on both code paths

#### Scenario: Extended types round-trip identically on both surfaces

- **GIVEN** schema `Booking` has `period: { "type": "calendar-range" }`
  (per `extended-field-types` REQ-EFT-002)
- **AND** a `Booking` exists with
  `period: { "start": "2026-05-01T09:00:00Z", "end": "2026-05-01T17:00:00Z" }`
- **WHEN** the Booking is fetched via REST `GET ‚Ä¶`
- **AND** the same Booking is fetched via GraphQL
  `query { booking(id: "<uuid>") { period { start end } } }`
- **THEN** both responses MUST carry the identical envelope
  `{ start, end }` with byte-identical ISO 8601 strings

#### Scenario: Nullability parity ‚Äî required fields non-nullable on both surfaces

- **GIVEN** schema `Order` has `required: ["orderNumber", "total"]`
- **WHEN** the GraphQL type `Order` is generated
- **THEN** `orderNumber` and `total` MUST be `Type::nonNull()` per
  the existing `graphql-api` Requirement "Generate mutations for a
  schema"
- **AND** when an `Order` is fetched via REST, the response MUST
  always include both fields (a stored Order that somehow lacks
  one is a data-integrity bug, not a missing-field render)

#### Scenario: CI parity test asserts shape equality across all schemas

- **GIVEN** the OR test suite
- **WHEN** the unified-OAS parity test runs
- **THEN** for every schema registered in OR, the test MUST:
  1. Pick (or seed) one object of that schema.
  2. Fetch it via REST.
  3. Fetch it via GraphQL with a query selecting every field
     declared in the schema.
  4. Assert field-by-field that names match, types match, values
     are equal under JSON canonicalisation.
- **AND** any divergence MUST fail the test with a diff showing
  which fields differ
- **AND** the test MUST also assert that any field returned by REST
  but missing from GraphQL (or vice versa) is a parity violation

### Requirement: A single MCP-discovery payload SHALL surface every schema's operations on both surfaces

This requirement extends both the existing `graphql-api` spec and
the existing `mcp-discovery` spec (per ADR-022's "MCP discovery"
abstraction). The MCP discovery endpoint MUST consume the unified
OAS document and emit one MCP tool per logical operation, marking
the transport (`rest` | `graphql` | `graphql-subscription`) so AI
agents know which surface to call. For schemas where REST and
GraphQL provide identical operations (the common case after this
delta), MCP MUST surface ONE tool per operation with both
transports listed, not two duplicate tools.

#### Scenario: MCP exposes one tool per operation with multi-transport metadata

- **GIVEN** schema `Order` has REST `GET /api/objects/commerce/order/{id}`
  and GraphQL `query order(id: ID!)` returning identical shape
- **WHEN** the MCP discovery endpoint is called
- **THEN** the discovered tool list MUST include ONE entry for
  "fetch one Order"
- **AND** that entry MUST carry
  `transports: ["rest", "graphql"]`
- **AND** MUST NOT contain two duplicate entries (one per surface)

#### Scenario: REST-only operations are surfaced with transports: ["rest"]

- **GIVEN** an operation `POST /api/objects/{register}/{schema}/import`
  that has no GraphQL equivalent (file upload not in Graph)
- **WHEN** the MCP discovery endpoint is called
- **THEN** the tool entry MUST carry `transports: ["rest"]` only
- **AND** the description MUST note the GraphQL gap so AI agents
  don't probe the wrong surface
