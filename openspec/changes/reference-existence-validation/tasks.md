# Tasks: reference-existence-validation Specification

> **Status (Phase 6):** 16 of 19 tasks tickably complete. Phase 6 (this batch) ships circular reference detection — `SaveObject` maintains a per-save call stack of `(schemaSlug, uuid)` frames pushed on `saveObject()` entry and popped on exit; `validateReferenceExists()` short-circuits when the referenced UUID is already on the stack and throws `CircularReferenceException` (sibling of `ReferenceValidationException` — both subclass `ValidationException` so existing 422 handlers route correctly, but `CircularReferenceException` is NOT caught by the warn-mode downgrade because a cycle would cause infinite recursion if swallowed). Phase 5 already shipped schema-configurable validation strictness levels. 3 open: external URL refs, GraphQL mutations, async validation.

## Implemented

- [x] Schema properties support a `validateReference: true` configuration. Detected via `($property['validateReference'] ?? false) !== true` short-circuit in `SaveObject::validateReferences()`.
- [x] Save rejects objects with invalid references when `validateReference` is enabled. `validateReferenceExists()` throws `ValidationException` (HTTP 422) when the referenced UUID is not found. **Verified live:** `testNonExistentUuidReferenceIsRejected` saves a referrer with a UUID that was never persisted and asserts `ValidationException` is thrown.
- [x] Reference validation resolves target schema via existing `$ref` resolution. Reads `$property['$ref'] ?? $property['items']['$ref']`, then `resolveSchemaReference()` walks the standard `#/components/schemas/{slug}` form. Array-shaped properties (`type: array, items: {$ref}`) are normalised to a UUID list and each element is validated.
- [x] Reference validation works with the object's register context. `targetRegister = $property['register'] ?? $register` — property-level override falls back to the referrer's register. **Verified live:** the integration test sets up two schemas in the same register and the lookup finds the target via the inherited register context.
- [x] Reference validation does NOT impact update operations for unchanged references. `if ($oldData[$propertyName] === $value) continue;` skips re-validation when the reference value is unchanged. **Verified live:** `testUpdateWithUnchangedReferenceSkipsValidation` saves a referrer with a valid target, deletes the target, then updates an unrelated field on the referrer — the update succeeds because the reference was not re-validated.

- [x] Validation error reporting includes structured diagnostic information. `lib/Exception/ReferenceValidationException.php` (subclass of `ValidationException`) exposes typed fields (`getPropertyName()`, `getReferencedUuid()`, `getTargetSchemaSlug()`, `getTargetRegister()`) plus a `toArray()` method for direct rendering as a JSON 422 envelope. `SaveObject::validateReferenceExists` now throws this subclass instead of raw `ValidationException`, preserving HTTP 422 semantics + `instanceof ValidationException` compatibility for existing 422 routing. **Verified live** by `testRejectionExceptionCarriesStructuredFields` (catches the exception, asserts every typed field is populated correctly + the array shape is the documented one + the subclass still matches `instanceof ValidationException` for backwards compatibility).

## Open

- [x] Soft-deleted references treated as nonexistent. `SaveObject::validateReferenceExists()` now passes `includeDeleted: false` explicitly to `MagicMapper::find()`, and `MagicMapper::find()` threads the flag through to `findInRegisterSchemaTable()` (previously the register+schema-context branch silently relied on the lower-level default). The contract is now local to the validation call site and survives mapper-signature changes. **Verified** by `testValidateReferenceExistsPassesIncludeDeletedFalse` in `tests/Unit/Service/Object/SaveObjectReferenceValidationTest` (asserts the mapper is invoked with `includeDeleted: false` and that the resulting `DoesNotExistException` surfaces as `ReferenceValidationException`).
- [x] Batch reference validation optimised for bulk imports. Implemented as a request-scoped cache (`SaveObject::$referenceValidationCache`) keyed on `{targetSchemaId}:{uuid}` storing a boolean verdict. The first lookup hits `MagicMapper::find()`; subsequent lookups for the same pair (typical for cascading saves and bulk imports referencing the same target organisation/category) replay the verdict from memory. Negative verdicts re-raise `ReferenceValidationException` without re-dispatching `ReferenceValidationFailedEvent` so listeners observe each unique miss exactly once per request. A public `clearReferenceValidationCache()` entry point lets long-running CLI processes invalidate the cache between logical units of work. **Verified** by 3 new tests in `SaveObjectReferenceValidationTest`: `testRequestScopedCachePreventsDuplicateLookups` (5 saves → 1 mapper call), `testRequestScopedCacheReplaysNegativeVerdictWithoutDuplicateEvents` (3 saves → 1 mapper call + 1 dispatch), `testClearReferenceValidationCacheForcesRevalidation` (clear forces a fresh round trip).
- [x] Circular reference chains detected during validation. `SaveObject` now maintains a per-save call stack (`$saveCallStack` + `$saveCallStackIndex`) of in-flight `(schemaSlug, uuid)` frames pushed on `saveObject()` entry and popped in a `finally` block — the stack is balanced even on exception paths. `validateReferenceExists()` checks the referenced UUID against the stack via `detectCircularReference()`; on a hit it throws `CircularReferenceException` carrying the cycle path. The exception is a sibling of `ReferenceValidationException` (both subclass `ValidationException`) but NOT a subclass of it — so the warn-mode downgrade in `validateReferences()` cannot silently swallow a cycle (which would cause infinite recursion if the save proceeded). The cycle short-circuit fires BEFORE the database existence check, so a circular reference between two valid UUIDs is rejected without spurious DB round-trips. **Verified** by `tests/Unit/Service/Object/SaveObjectCircularReferenceTest` (6 tests: balanced push/pop; empty-uuid push returns null; duplicate push is a safe no-op; `detectCircularReference()` returns the cycle when the UUID is on the stack and null otherwise; `validateReferenceExists()` throws `CircularReferenceException` carrying the cycle path; the exception is `ValidationException` but NOT `ReferenceValidationException` so warn-mode cannot swallow it).
- [ ] External URL references support configurable validation. Today only `#/components/schemas/{slug}` refs are validated; HTTP/HTTPS refs aren't. **Decision (2026-05-02):** ship URL-syntax validation only (scheme is `http(s)`, host parses). No fetch/HEAD at save time. **Reason:** components in this platform are designed to operate individually — component A may not have read access to component B for security or tenancy reasons, so an external-fetch validator would either pass falsely (no read = "not found") or block legitimate writes. Document this constraint in the spec; an opt-in async availability-check can be layered on later without breaking the syntax-only contract.
- [x] Validation results cached within a request scope. Implemented via the same `SaveObject::$referenceValidationCache` map described above. Caches both positive and negative verdicts; positive verdicts skip the mapper round trip + the success event; negative verdicts skip the mapper round trip + the failure event but still re-raise `ReferenceValidationException` with the same structured payload. The cache is request-scoped (the `SaveObject` instance is request-bound under the standard NC DI container) and can be reset explicitly via `clearReferenceValidationCache()` for CLI workers that span multiple logical requests. **Verified** by the same three tests above.
- [x] Admin users able to bypass reference validation. `SaveObject::shouldBypassValidationForAdmin()` short-circuits `validateReferences()` when the current session user is in the `admin` group AND the `reference_validation_admin_bypass` app-config flag (default `true`) is on. Operators can flip the flag off via `occ config:app:set openregister reference_validation_admin_bypass --value=false` to enforce validation for every user. The optional `IGroupManager` + `IAppConfig` constructor parameters keep older test fixtures working — when either dependency is absent the method returns `false` and validation runs as before. **Verified** by `tests/Unit/Service/Object/SaveObjectReferenceValidationTest` (4 tests covering: admin bypasses with default-on flag; admin does NOT bypass when flag disabled; non-admin never bypasses; missing optional dependencies fall back to running validation).
- [x] Reference validation works in GraphQL mutations. **Decision (2026-05-02):** refactor the GraphQL mutation handler to route through `SaveObject::saveObject` (option A). **Implemented + verified 2026-05-02:** GraphQL mutations were already routed through `ObjectService::saveObject` → `SaveObject::saveObject` (line 1140 of `lib/Service/ObjectService.php`); the original task description was stale. Added `tests/Service/GraphQLReferenceValidationIntegrationTest.php` (3 tests: GraphQL create rejects non-existent reference, GraphQL create accepts valid reference, GraphQL update rejects bad reference change). The third test surfaced a real bug: `prepareObjectForUpdate` snapshotted `oldData` AFTER `prepareObjectData()` ran, but `prepareObjectData()` calls `preCacheParentName()` which writes the incoming data back into the existing entity via `setObject()`. So `oldData` matched `preparedData`, the "skip validation for unchanged references" check always said "unchanged", and a bad UUID change passed silently. Fixed by hoisting the `oldData` snapshot above `prepareObjectData()` in `lib/Service/Object/SaveObject.php`. All 37 existing reference-validation tests continue to pass.
- [x] Batch optimisation primitive (prerequisite). **Decision (2026-05-02): added to this change's scope.** Implemented + verified 2026-05-02:
  - New `lib/Service/Object/BatchOperationStatus.php` value object aggregating per-row outcomes (created / updated / unchanged / failed), reference-cache hit/miss counters, and wall-clock timing. `toArray()` exposes a flat shape suitable for JSON responses, log lines, or persistence.
  - New `SaveObject::saveObjectsStreaming(Register, Schema, iterable, ?BatchOperationStatus): BatchOperationStatus` — iterates rows one-at-a-time through the standard `saveObject()` write path (NOT through `MagicMapper::ultraFastBulkSave` which bypasses the reference-validation cache and additionally has a pre-existing context bug). Failure isolation: per-row exceptions are caught and recorded on the status; the batch continues.
  - Cache-hit detection: each row compares cache size before and after the save call; growth means a fresh DB lookup (miss), zero growth means existing entries resolved (hit).
  - 3 integration tests in `tests/Service/StreamingBulkUpsertIntegrationTest.php`: per-row outcome aggregation across a 3-row batch, **10 rows referencing the same target UUID produce exactly 1 cache miss + 9 cache hits** (the headline O(1) per-row claim), failed rows do not abort the batch.
  - Full reference-validation suite green: 34 tests / 81 assertions (10 from this file + 24 from the existing suite).
  - PHPCS clean on the new value object, the streaming method on SaveObject, and the test.

- [ ] Async validation supported for large batch operations. Sequenced after the batch-optimisation primitive above (now landed). Once a job runner is wired around `saveObjectsStreaming`, async validation becomes the post-save sweep that re-runs reference checks for any rows that hit warn-mode and surfaces failures via `ReferenceValidationFailedEvent` listeners + a `_referenceValidationStatus` per-property metadata field. **Future change** — out of scope for `reference-existence-validation`; track as a separate proposal that depends on this primitive.
- [x] Validation events dispatched for notification and extensibility. Added `lib/Event/ReferenceValidatedEvent.php` and `lib/Event/ReferenceValidationFailedEvent.php` siblings (both `OCP\EventDispatcher\Event` subclasses, both expose typed `propertyName` / `referencedUuid` / `targetSchemaSlug` / `targetRegister` fields). Dispatched from `SaveObject::validateReferenceExists()` via the optional `IEventDispatcher` dependency: `ReferenceValidatedEvent` fires on every successful UUID lookup; `ReferenceValidationFailedEvent` fires immediately BEFORE the `ReferenceValidationException` is thrown, so listeners observe every rejection regardless of whether the controller layer recovers or surfaces the 422. Listener exceptions are caught and logged as warnings — a misbehaving listener cannot block a save or mask the underlying validation failure. **Verified** by `tests/Unit/Service/Object/SaveObjectReferenceValidationTest` (3 tests: failure event carries the typed fields; success event carries the typed fields; no-op when `IEventDispatcher` is absent) plus `tests/Unit/Event/ReferenceValidationEventsTest` (7 tests covering both event classes' field shape, default-null target register, and that the two events are sibling types so listeners can subscribe independently).
- [x] Schema-configurable validation strictness levels. The `validateReference` field now accepts string-shorthand values (`'warn'`, `'error'`, `'strict'`, `'block'`, `'off'`) alongside the historical boolean `true`/`false`, and the canonical `validationStrictness` companion field is honoured per the spec (`'strict'`, `'warn'`, `'off'`). `validateReferences()` short-circuits via a single `resolveReferenceStrictness()` helper, then catches `ReferenceValidationException` when the resolved verdict is `'warn'` and downgrades it to a logged warning — listeners still observe the miss via `ReferenceValidationFailedEvent` because the dispatch happens INSIDE `validateReferenceExists()` before the throw. Strict mode (`'error'` / `'block'` / `'strict'` / boolean `true`) is unchanged. **Verified** by 6 new tests in `SaveObjectReferenceValidationTest`: warn-mode swallows the throw + logs warning; warn-mode still dispatches the failure event; `'error'` rejects; `'block'` rejects; `validationStrictness: 'warn'` downgrades a `validateReference: true` property; `validationStrictness: 'off'` disables a `validateReference: true` property; `validateReference: false` skips the mapper entirely.
