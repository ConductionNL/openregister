# OAS Validation Specification

## Why

The OpenAPI spec generated by `OasService::createOas()` is consumed by Swagger UI, generated SDKs, and reverse proxies, but earlier versions emitted documents that broke each of those tools: dangling `$ref`, duplicate `operationId`s across registers, orphan tags, relative server URLs, and `datetime` types that aren't part of the OpenAPI 3.1.0 vocabulary. NL government API Design Rules (API-01 method whitelist, API-03 status code whitelist, API-46 problem details) also require structural enforcement we previously had to do by hand. A first-class validation layer on the generation path catches these issues at build time, auto-corrects the safe ones, and lets operators run in `strict` mode when they want CI to fail loudly.

## What Changes

- Extend `OasService::validateOasIntegrity()` with four new passes: `validateServerUrls()`, `validateOperationIdUniqueness()`, `validateTagConsistency()`, and `validateNlGovRules()`.
- Auto-correct safe violations: dangling `$ref` → `type: string`, duplicate `operationId` → numeric suffix (`Foo`, `Foo_2`, `Foo_3` …), orphan tag → auto-injected with a generated description.
- Add `OasValidationReport` value object collecting every issue with a JSON Pointer path (e.g. `paths./objects/zaken/meldingen.get.responses.200`), a stable machine code (`OasValidationReport::CODE_*`), and a severity (`error`/`warning`/`auto_corrected`).
- Add `OasValidationException` carrying the report; thrown in strict mode when any `error`-severity issue is found.
- Add `?strict=true` query parameter on `OasController` → HTTP 422 with the report attached when validation fails.
- Add `?validate=true` query parameter on `OasController` → attaches an `x-validation-summary` extension to the response.
- Log issues via `LoggerInterface::warning|error()` when a logger is injected.
- Expose the report through `OasService::getLastValidationReport()` for callers that want to inspect without throwing.
- Add request/response runtime validation against the generated OAS using `opis/json-schema`, gated on the wider request-validation middleware design.
- Add ETag/cache layer for OAS responses so heavy registers don't regenerate the spec on every fetch.
- Add Redocly CLI lint integration (`npx @redocly/cli lint`) in CI once a stable test register fixture exists.
- Add schema-validation-on-import in `ImportHandler` so imported schemas are pre-checked for OAS compatibility.
- Add API-46 RFC 7807 Problem Details enrichment to the `Error` schema in `BaseOas.json` (`type`, `title`, `status`, `detail`, `instance`).
- Add OpenAPI 3.1.0 meta-schema validation via `opis/json-schema` for the broadest structural check, gated on vendoring or fetching the meta-schema document.

## Problem
Ensure that the OpenAPI Specification (OAS) output generated by `OasService::createOas()` is structurally valid, semantically correct, and compliant with OpenAPI 3.1.0, NL API Design Rules, and Redocly lint rules. This spec covers request/response validation against the generated OAS, schema validation on import, OAS integrity checks, automated compliance testing, validation error reporting, validation modes, performance considerations, and CI integration.
